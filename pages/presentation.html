03 — Base de données

Structure d'un ticket

Chaque ticket est un objet JSON. Tous les tickets d'un utilisateur sont stockés dans un seul tableau.

Schéma objet Ticket

ChampTypeValeurs
id PKnumberDate.now()
titlestringTexte libre
categorystringTechnique Facturation Compte
prioritystringbas moyen eleve critique
statusstringouvert en_cours resolu
datestringYYYY-MM-DD
descstringDescription
messagesarray[{from, text, time}]
💬 Objet Message (dans messages[])
ChampValeurs
from"agent" ou "user"
textContenu du message
timeFormat HH:MM

Exemple JSON réel

{ "id": 1712345678901, "title": "Erreur lors du paiement", "category": "Technique", "priority": "critique", "status": "en_cours", "date": "2026-04-05", "desc": "Une erreur 500 survient...", "messages": [ { "from": "agent", "text": "Bonjour, ticket pris en charge.", "time": "10:30" } ] }

04 — Stockage

localStorage — La "base de données"

Toutes les données sont persistées dans le navigateur sous forme de paires clé/valeur JSON.

Clés utilisées

tf_user
Session active — { email, name, role, avatar }
Supprimé à la déconnexion
tf_tickets_<email>
Tableau JSON des tickets d'un utilisateur.
Ex: tf_tickets_user@ticketflow.fr
tf_admin_users
Liste des comptes utilisateurs gérés par l'admin
tf_admin_agents
Liste des agents de support
tf_features
Modules actifs/inactifs — { "dashboard_agent": false }
⚠️ Limites
  • Non partagé entre navigateurs / appareils
  • Capacité ~5 Mo par domaine
  • Authentification simulée (pas de vrai backend)
