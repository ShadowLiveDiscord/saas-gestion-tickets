Architecture, base de données tickets et cycle de mise à jour — Projet SaaS de gestion de tickets support.

01 — Architecture

Architecture du projet

TicketFlow est une application 100 % front-end — aucun serveur, aucune base de données distante.

🌐

Navigateur

Toute la logique s'exécute dans le navigateur de l'utilisateur

⚙️

JavaScript Vanilla

ES6+ pur — aucun framework (React, Vue…)

💾

localStorage

Persistance des données côté client (tickets, users, flags)

📁 Structure des fichiers
TicketFlow/ ├── index.html ← Landing page publique ├── cahier-des-charges.html ← Spécifications ├── css/ │ ├── style.css ← Variables, boutons, cartes, formulaires │ ├── auth.css ← Pages login / register │ └── dashboard.css ← Layout sidebar, topbar, tables ├── js/ │ └── features.js ← Système de feature flags (modules on/off) └── pages/ ├── login.html ← Authentification (simulée) ├── register.html ← Inscription ├── dashboard-user.html ← Espace utilisateur ├── dashboard-agent.html ← Espace agent ├── dashboard-admin.html ← Panel administrateur ├── codeur-generator.html ← Générateur message + devis PDF └── unavailable.html ← Page module désactivé
🔄 Flux d'une session utilisateur
1

Accès à la page de connexion

login.html vérifie si le module login est actif via requireFeature('login')

2

Authentification

Vérification de l'email + mot de passe contre les comptes de démo. La session est stockée dans localStorage['tf_user']

3

Redirection selon le rôle

admin → dashboard-admin.html  |  agent → dashboard-agent.html  |  user → dashboard-user.html

4

Chargement des données

Les tickets sont lus depuis localStorage['tf_tickets_<email>'] et affichés dynamiquement en JS

5

Interactions & persistance

Toute modification (création, statut, message) est immédiatement écrite dans le localStorage

02 — Rôles

Rôles & accès

3 rôles distincts avec des permissions différentes, définis à la connexion.

🔴 Administrateur

  • Vue KPIs globaux
  • Tous les tickets
  • Gestion utilisateurs
  • Gestion agents
  • Activation modules (feature flags)
  • Paramètres système
  • Accès garanti même si tous les modules sont désactivés

🟢 Agent

  • Voir tous les tickets
  • Répondre aux clients
  • Changer statut & priorité
  • Onglet "mes tickets assignés"

🔵 Utilisateur

  • Créer des tickets
  • Voir ses propres tickets
  • Filtrer & rechercher
  • Messagerie avec l'agent
  • Modifier son profil

03 — Base de données

Structure d'un ticket

Chaque ticket est un objet JSON stocké dans un tableau dans le localStorage.

📋 Schéma d'un objet Ticket
ChampTypeValeurs possiblesDescription
id PK number Date.now() à la création Identifiant unique du ticket
title string Texte libre Titre / résumé du problème
category string Technique Facturation Compte Commercial Catégorie du ticket
priority string bas moyen eleve critique Niveau de priorité
status string ouvert en_cours resolu ferme État actuel du ticket
date string Format YYYY-MM-DD Date de création
desc string Texte libre Description détaillée du problème
messages array [ { from, text, time } ] Fil de messages entre user et agent
userEmail string email de l'auteur Propriétaire du ticket (côté agent)
💾 Exemple JSON d'un ticket
{ "id": 1712345678901, "title": "Impossible de me connecter à mon compte", "category": "Compte", "priority": "eleve", "status": "en_cours", "date": "2026-04-05", "desc": "Depuis ce matin, impossible d'accéder à mon compte...", "messages": [ { "from": "agent", "text": "Bonjour, nous avons pris en charge votre ticket.", "time": "10:30" } ] }
📨 Schéma d'un objet Message (dans messages[])
ChampTypeValeursDescription
fromstring"agent" ou "user"Expéditeur du message
textstringTexte libreContenu du message
timestringFormat HH:MMHeure d'envoi
04 — Stockage

Stockage localStorage

Toutes les données sont persistées dans le localStorage du navigateur via des clés JSON.

🗂 Clés localStorage utilisées
tf_user
Session active — objet { email, name, role, avatar }. Supprimé à la déconnexion.
tf_tickets_<email>
Tableau JSON des tickets d'un utilisateur. Une clé par compte (ex: tf_tickets_user@ticketflow.fr).
tf_admin_users
Liste des comptes utilisateurs gérés par l'admin. Tableau JSON d'objets { id, name, email, role, createdAt }.
tf_admin_agents
Liste des agents. Tableau JSON d'objets { id, name, email, specialty, tickets, resolved }.
tf_features
Carte des modules activés/désactivés. Objet JSON { moduleKey: boolean }. Ex: { "dashboard_agent": false }.
⚠️ Limites du stockage client
  • Les données ne sont pas partagées entre navigateurs ni entre appareils
  • Capacité limitée à ~5 Mo par domaine
  • L'authentification est simulée (pas de vrai backend)
  • Un vrai projet en production utiliserait une base de données SQL/NoSQL + API REST
05 — Mise à jour

Cycle de vie & mise à jour d'un ticket

Un ticket passe par plusieurs états. Agents et utilisateurs peuvent interagir dessus.

🔄 Cycle des statuts
Ouvert
Création
→
En cours
Agent répond
→
Résolu
Problème réglé
→
Fermé
Archivé

⚡ Transition automatique : quand un agent envoie un premier message, le statut passe automatiquement de Ouvert → En cours.

➕ 1. Création d'un ticket (côté utilisateur)
1

L'utilisateur remplit le formulaire

Titre, catégorie, priorité, description. Les 3 premiers champs sont obligatoires.

2

Appel à submitNewTicket()

Création de l'objet ticket avec id: Date.now(), status: 'ouvert', messages: []

3

Ajout en tête de tableau + sauvegarde

tickets.unshift(newTicket) → saveTickets() → localStorage.setItem('tf_tickets_<email>', JSON.stringify(tickets))

4

Rafraîchissement de l'interface

renderTickets() et updateStats() mettent à jour l'affichage instantanément

// dashboard-user.html — submitNewTicket() const newTicket = { id: Date.now(), title, category, priority, desc, status: 'ouvert', date: new Date().toISOString().split('T')[0], messages: [] }; tickets.unshift(newTicket); saveTickets(); // → localStorage renderTickets(); // → re-render du tableau updateStats(); // → badges & compteurs
✏️ 2. Mise à jour du statut (côté agent)
1

L'agent clique sur "Traiter" sur un ticket

Appel à openTicket(id) → ouverture de la modale avec les détails

2

L'agent change le statut via le <select>

Appel à changeStatus() → trouve le ticket par id dans allTickets[], modifie t.status

3

Mise à jour de l'affichage

updateStats() et filterAgentTickets() re-rendent le tableau immédiatement

// dashboard-agent.html — changeStatus() function changeStatus() { const t = allTickets.find(x => x.id === currentTicketId); t.status = document.getElementById('modal-status-select').value; updateStats(); filterAgentTickets(); }
💬 3. Réponse d'un agent (messagerie)
1

L'agent saisit un message et clique "Envoyer"

Appel à agentReply()

2

Ajout du message dans t.messages[]

Objet { from: 'agent', text, time } poussé dans le tableau

3

Transition automatique de statut

Si le ticket était 'ouvert', il passe automatiquement à 'en_cours'

// dashboard-agent.html — agentReply() const time = now.getHours() + ':' + String(now.getMinutes()).padStart(2, '0'); t.messages.push({ from: 'agent', text, time }); // Transition automatique ouvert → en_cours if (t.status === 'ouvert') { t.status = 'en_cours'; updateStats(); }
06 — Feature Flags

Système de Feature Flags

L'admin peut activer ou désactiver chaque module du site en temps réel, sans rechargement.

🔀 Principe de fonctionnement
// js/features.js // Vérifier si un module est actif isEnabled('user_create_ticket') // → true / false // Rediriger vers "Indisponible" si module désactivé requireFeature('dashboard_user') // Activer/désactiver un module et persister toggleFeatureSave('agent_reply', false) // Désactiver TOUS les modules sauf login et dashboard_admin setAllFeaturesSave(false)

⚠️ Les modules login et dashboard_admin sont verrouillés — ils ne peuvent jamais être désactivés pour garantir l'accès de l'administrateur.

07 — Outils

Outils intégrés

Module additionnel intégré nativement dans le projet.

✉ Générateur Codeur.com + Devis PDF

✉ Message de candidature

  • 4 tons de message (Pro, Chaleureux, Concis, Expert)
  • Génération instantanée en temps réel
  • Copie en 1 clic dans le presse-papiers

📄 Devis PDF automatique

  • Lignes de prestation dynamiques
  • Calcul HT / TVA / TTC automatique
  • Aperçu en temps réel
  • Export PDF via jsPDF (sans serveur)

Accessible depuis la sidebar de tous les dashboards. Peut être désactivé via le feature flag codeur_generator.
